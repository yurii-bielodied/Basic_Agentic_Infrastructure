# MCP Sampling, Elicitation та MCP Apps: практичні технічні й бізнесові кейси

## Вступ

Model Context Protocol (MCP) — це відкритий протокол для підключення AI-застосунків до зовнішніх систем, даних та інструментів. У базовій моделі MCP сервери надають tools, resources і prompts, а клієнти можуть підтримувати додаткові можливості, зокрема Sampling та Elicitation. MCP Apps — це офіційне розширення MCP для вбудованих інтерактивних інтерфейсів усередині хоста.

Цей документ фокусується на трьох практичних блоках:

1. **Sampling** — коли серверу потрібно отримати генерацію або reasoning від моделі через клієнт.
2. **Elicitation** — коли серверу потрібно динамічно добрати від користувача бракуючі структуровані дані.
3. **MCP Apps** — коли текстової відповіді вже недостатньо і потрібен інтерактивний UI прямо в розмові.

---

## 1. Sampling

### Що таке Sampling

Sampling у MCP дозволяє серверу запитувати completion або іншу модельну взаємодію через MCP-клієнт, а не напряму через SDK конкретного провайдера. Це означає, що сервер може залишатися модельно-нейтральним: він не зобов’язаний напряму інтегрувати OpenAI, Anthropic чи іншу LLM-платформу, не повинен зберігати власні API-ключі та може покладатися на політики доступу, які вже контролюються клієнтом.

Практично Sampling корисний тоді, коли LLM-виклик є частиною більшого workflow. Сервер спочатку збирає факти через tools або resources, а потім делегує моделі задачу: підсумувати, пояснити, знайти ризики, побудувати план дій або сформувати відповідь для користувача.

### Технічна цінність Sampling

З технічної точки зору Sampling дає кілька важливих переваг:

- **Модельна незалежність**: MCP-серверу не потрібно вбудовувати окремий код під кожного LLM-провайдера.
- **Кращий контроль безпеки**: право на запуск моделі залишається на боці клієнта.
- **Природна агентна поведінка**: сервер може запускати вкладені LLM-виклики як частину складнішої логіки.
- **Простіша архітектура**: один сервер може працювати з різними MCP-хостами без жорсткої прив’язки до конкретного AI API.

### Реальні технічні кейси Sampling

#### 1. AI-самаризація інцидентів

MCP-сервер отримує логи, події Kubernetes, Prometheus alerts, помилки з CI/CD або результати health-check’ів. Далі через Sampling він просить модель згенерувати короткий incident summary:

- що сталося;
- які найімовірніші причини;
- що перевірити далі;
- які дії варто виконати першими.

Такий сценарій особливо корисний для SRE, DevOps та platform teams, тому що модель не замінює observability-інструменти, а допомагає швидше інтерпретувати вже зібрані технічні сигнали.

#### 2. Пояснення Terraform plan або deployment diff

Сервер читає зміни між двома ревізіями Terraform, Helm values або Kubernetes manifest’ів, а далі через Sampling просить модель пояснити ці зміни людською мовою:

- що саме змінюється;
- які ресурси або сервіси можуть бути зачеплені;
- які є ризики;
- що треба перевірити після застосування.

Це корисно для code review, change review, погодження релізів і комунікації між інженерними та бізнес-командами.

#### 3. Генерація runbook-рекомендацій

Після збору поточного стану системи через tools сервер може попросити модель сформувати короткий план дій: які перевірки зробити першими, які команди запустити, які метрики або події переглянути. Це перетворює “сирі” технічні дані на керований, послідовний workflow для інженера.

#### 4. Пояснення складних конфігурацій або помилок

Якщо сервер вміє читати конфігурації, логи або помилки з інфраструктури, Sampling можна використати для генерації пояснення на зрозумілішому рівні: що означає помилка, чому вона могла виникнути, які є типові варіанти виправлення.

### Бізнесова цінність Sampling

З бізнесової точки зору Sampling дає:

- швидший первинний аналіз проблем;
- зменшення часу на ручне читання логів і diff’ів;
- кращу комунікацію між технічними та нетехнічними ролями;
- потенційне скорочення MTTR завдяки швидшому формуванню гіпотез і наступних кроків;
- централізованіший контроль доступу до LLM та витрат на моделі.

### Коли Sampling особливо доречний

Sampling найкраще працює там, де:

- є багато технічного контексту, який уже доступний через tools/resources;
- потрібно отримати короткий, зрозумілий або пояснювальний результат;
- серверу потрібен reasoning-шар, але без прямої інтеграції з конкретним провайдером моделі.

## 2. Elicitation

### Що таке Elicitation

Elicitation у MCP дозволяє серверу під час виконання сценарію запросити в користувача додаткові дані через клієнт. На практиці це не просто “поставити уточнювальне запитання”, а отримати **структуровану відповідь**, зазвичай на основі JSON Schema.

Це дуже важливо для сценаріїв, де користувач хоче виконати дію, але ще не надав усі необхідні параметри. Замість довгого неструктурованого діалогу сервер може запросити лише ті поля, яких бракує, і отримати валідовану відповідь.

### Технічна цінність Elicitation

Elicitation вирішує поширену проблему багатьох агентних workflow: користувач формулює задачу загально, але для її виконання потрібні конкретні параметри.

Переваги такого підходу:

- **структурований ввід** замість хаотичного текстового діалогу;
- **валідація полів** на рівні схеми;
- **зменшення неоднозначності**;
- **повторюваність процесу** для однакових типів задач;
- **кращий UX** для internal tools і self-service платформ.

### Реальні технічні кейси Elicitation

#### 1. Change request / deployment request

Користувач пише: “Онови сервіс” або “Розгорни нову версію”.

Але для дії бракує даних: cluster, namespace, deployment, image tag, replicas, maintenance window, rollback policy, change reason. Замість багатоходового чату сервер запускає Elicitation і просить рівно ті поля, яких не вистачає.

Це зручно для Kubernetes, Terraform, CI/CD, database changes і будь-якого керованого change management процесу.

#### 2. Створення нового сервісу або середовища

Якщо користувач хоче створити новий сервіс, сервер може добрати:

- owner/team;
- repo URL;
- environment;
- CPU/RAM limits;
- requirements до ingress;
- observability settings;
- потрібні secrets або інтеграції.

У результаті self-service платформа отримує повні й валідні дані для подальшої автоматизації.

#### 3. Support intake / service desk

Користувач пише: “У мене не працює застосунок”.

Через Elicitation сервер може зібрати:

- environment;
- affected service;
- час проблеми;
- severity;
- steps to reproduce;
- чи проблема відтворюється постійно.

Це дозволяє стандартизувати вхідні запити і зменшує кількість додаткових уточнень від support-команди.

#### 4. Погодження або ручне підтвердження дії

У складніших workflow Elicitation можна використовувати не лише для технічних полів, а й для підтвердження наміру: причина зміни, бізнес-контекст, підтвердження ризику, вибір вікна обслуговування. Це добре працює там, де потрібен human-in-the-loop перед потенційно ризиковою дією.

### Бізнесова цінність Elicitation

Для бізнесу Elicitation означає:

- менше неповних або помилкових заявок;
- кращу стандартизацію change management;
- швидший onboarding до внутрішніх платформ;
- менше ручного back-and-forth між користувачем і підтримкою;
- вищу якість вхідних даних для автоматизації.

### Обмеження та важливі зауваження

Elicitation добре підходить для збору **несекретної** та чітко структурованої інформації. Його не варто розглядати як механізм для запиту чутливих даних. Також він найкраще працює в сценаріях, де поля можна описати через схему й де важлива валідність значень.

## 3. MCP Apps

### Що таке MCP Apps

MCP Apps — це розширення MCP, яке дозволяє серверу повертати не лише текст або структуровані дані, а й **інтерактивний UI**, який хост рендерить прямо всередині розмови. Тобто замість звичайної текстової відповіді користувач може побачити форму, дашборд, таблицю, графік, PDF viewer або інший інтерфейс, з яким можна взаємодіяти.

MCP Apps перетворюють MCP з “протоколу для tools” на основу для побудови повноцінних AI-assisted mini-apps у межах одного хоста.

### Архітектурна ідея MCP Apps

Базовий патерн MCP Apps складається з двох частин:

1. **Tool**, який у своєму описі оголошує UI resource.
2. **UI resource**, який віддається сервером і рендериться хостом як HTML-інтерфейс.

Хост відкриває цей інтерфейс у sandboxed iframe та забезпечує взаємодію між UI і сервером. Це означає, що MCP App працює ізольовано, але може брати участь у розмові та викликати логіку MCP-сервера.

### Технічна цінність MCP Apps

MCP Apps особливо корисні там, де текст уже не дає хорошого UX. Наприклад:

- коли є багато полів і залежностей між ними;
- коли потрібно показати статуси, таблиці або графіки;
- коли користувач має пройти покроковий workflow;
- коли потрібно переглядати документи, артефакти або складні дані inline;
- коли потрібні дії кнопками, фільтрами, сортуванням, drill-down або перемикачами.

### Реальні технічні кейси MCP Apps

#### 1. Observability dashboard

Замість відповіді в стилі “система працює нормально” MCP App може показати:

- статус сервісів;
- графіки помилок і затримок;
- останні alerts;
- проблемні pod’и;
- drill-down по namespace або сервісу.

Такий підхід значно ближчий до реального робочого UX для SRE та platform teams, ніж чистий текст у чаті.

#### 2. Configuration wizard / deployment assistant

Якщо користувач хоче змінити deployment, створити сервіс, налаштувати ingress або оновити параметри, MCP App може показати інтерактивну форму з валідацією, defaults і логікою залежностей між полями. Це особливо корисно для production/staging сценаріїв, де набір параметрів відрізняється залежно від контексту.

#### 3. Inline review для документів, policy або PDF

MCP App може відкрити документ прямо в розмові та дозволити користувачу переглядати його, переходити між секціями, бачити виділені фрагменти й приймати рішення без перемикання в інші вкладки або системи.

#### 4. Data exploration і бізнес-аналітика

Для бізнесових сценаріїв MCP Apps підходять для:

- cohort/segment analysis;
- budget allocation;
- what-if scenario modeling;
- customer segmentation;
- exploratory dashboards.

Тобто MCP Apps — це не лише DevOps/UI для інженерів, а й інструмент для інтерактивної роботи з бізнес-даними.

### Бізнесова цінність MCP Apps

З бізнесової точки зору MCP Apps дають:

- значно кращий UX порівняно з довгим текстовим діалогом;
- менше переключень між різними інструментами;
- кращу зрозумілість стану системи або параметрів дії;
- швидше проходження multi-step workflow;
- природніший формат для внутрішніх платформ самообслуговування.

### Чому MCP Apps особливо сильні в enterprise-кейсах

Внутрішні платформи, support-системи, change management, аналітичні панелі та review-процеси часто вимагають не просто “відповіді моделі”, а інтерфейсу, з яким можна взаємодіяти. MCP Apps заповнюють саме цю нішу: вони поєднують conversational experience з можливостями вбудованого продуктового UI.

## Порівняння трьох підходів

| Механізм | Головне призначення | Найкраще підходить для |
|---|---|---|
| **Sampling** | Отримати генерацію або reasoning від моделі через клієнт | Самаризація, пояснення, аналіз, рекомендації |
| **Elicitation** | Добрати від користувача відсутні структуровані дані | Форми, уточнення параметрів, intake-процеси |
| **MCP Apps** | Дати інтерактивний UI прямо в розмові | Дашборди, конфігуратори, review, multi-step workflows |

## Практичний висновок

Найбільшу цінність ці можливості дають не окремо, а в комбінації.

Типовий сильний сценарій виглядає так:

1. **MCP App** відкриває інтерактивний інтерфейс.
2. **Elicitation** добирає бракуючі поля, якщо користувач не вказав їх одразу.
3. **Tools** читають фактичний стан зовнішньої системи.
4. **Sampling** формує підсумок, ризики, пояснення або рекомендації.

Саме така зв’язка найбільш практична для реальних внутрішніх платформ, DevOps, support, аналітики та self-service workflow.

---

## Висновок

Sampling, Elicitation і MCP Apps — це три різні рівні взаємодії в екосистемі MCP.

- **Sampling** додає серверу доступ до LLM через клієнт для генерації, пояснення та reasoning.
- **Elicitation** дозволяє серверу збирати від користувача структуровані й валідовані дані прямо під час workflow.
- **MCP Apps** додають інтерактивний UI всередині розмови й роблять можливими сценарії, де текст уже недостатній.

Для практичних технічних та бізнесових кейсів це означає одне: MCP дозволяє будувати не просто “чат із тулзами”, а повноцінні AI-assisted workflow з контрольованим UX, структурованими даними та вбудованими інтерфейсами.
